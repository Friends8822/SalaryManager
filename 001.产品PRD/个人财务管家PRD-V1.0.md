## 1. 产品概述

### 1.1 一句话定位

> **财务管家** — 3 秒记一笔，一张图看懂你的钱去哪了。

### 1.2 产品愿景

成为 18-35 岁年轻用户记账的首选工具——不是因为功能最全，而是因为「坚持得下去」。

### 1.3 产品背景

#### 1.3.1 行业现状

- 中国移动支付渗透率超过 95%，但微信/支付宝的账单功能分散、不互通
- 用户平均同时在 3-4 个支付渠道产生消费（微信、支付宝、银行卡、信用卡）
- 记账类 App 下载量年增长 15%，但 **30 日留存率普遍低于 15%**——大多数人记着记着就放弃了

#### 1.3.2 核心问题

| 问题 | 表现 | 根因 |
|------|------|------|
| 记账坚持不下去 | 30 日留存 < 15% | 操作繁琐，每笔记账需要 6-10 步 |
| 统计看不懂 | 用户只看总额不看分析 | 图表太专业，缺少一眼就懂的洞察 |
| 分类焦虑 | 用户纠结选哪个分类 | 分类太多太细，选择困难 |
| 数据孤岛 | 各支付渠道账单分散 | 缺少统一汇总工具 |

#### 1.3.3 我们的解法

| 问题 | 解法 | 核心策略 |
|------|------|---------|
| 坚持不下去 | 极简化记账：3 步、5 秒内完成 | **减少摩擦 > 增加功能** |
| 统计看不懂 | 首页 Dashboard 一张图讲清本月消费 | **视觉化 > 数据罗列** |
| 分类焦虑 | 仅 8 个默认分类，智能记忆上次选择 | **少即是多** |
| 数据孤岛 | 手动记录（V1.0）→ 自动同步（V2.0） | **先培养习惯，再解决效率** |

### 1.4 核心价值主张

```
┌────────────────────────────────────────────┐
│                                            │
│    📱 打开 App                              │
│         ↓                                  │
│    👆 输入金额 → 选分类 → 保存               │
│         ↓                                  │
│    ✅ 不到 5 秒，完成记账                    │
│         ↓                                  │
│    📊 首页自动更新：本月花了多少、花在哪       │
│                                            │
└────────────────────────────────────────────┘
```

### 1.5 产品原则

以下是指导 V1.0 所有决策的核心原则，当需求冲突时，按优先级排序：

| 优先级 | 原则 | 说明 |
|--------|------|------|
| P0 | **速度优先** | 任何让记账变慢的功能都不做。单次记账 > 5 秒 = 失败 |
| P0 | **一眼就懂** | 首页不需要滚动就能获取核心信息（总额、分类、趋势） |
| P1 | **离线可用** | 不依赖网络，所有数据本地存储，打开即用 |
| P1 | **安全感** | 数据可导出、可备份，用户完全掌控自己的数据 |
| P2 | **美观舒适** | 春日绿色主题，柔和护眼，不造成视觉疲劳 |

## 2. 市场分析与竞品研究

### 2.1 市场规模（TAM / SAM / SOM）

| 层级 | 定义 | 估算 |
|------|------|------|
| TAM（总可寻址市场） | 中国有记账需求的移动互联网用户 | ~2.5 亿人 |
| SAM（可服务市场） | 18-40 岁、使用智能手机、愿意尝试记账工具 | ~8,000 万人 |
| SOM（可获取市场） | V1.0 上线后一年内可触达的用户 | ~5-10 万人 |

### 2.2 竞品深度分析

#### 2.2.1 竞品矩阵

| 维度 | 随手记 | 鲨鱼记账 | 钱迹 | 喵喵记账 | **财务管家（我们）** |
|------|--------|---------|------|---------|-------------------|
| 定位 | 专业全面 | 极简记账 | 轻量记账 | 游戏化记账 | **春日极简记账** |
| 记账速度 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 统计报表 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 学习成本 | 高 | 低 | 低 | 中 | 极低 |
| 多账户 | ✅ 全支持 | ❌ 无 | ✅ 支持 | ❌ 无 | P2 规划 |
| 预算功能 | ✅ | ❌ | ✅ | ✅ | ✅ P0 |
| 数据导出 | ✅ 多格式 | ❌ | ✅ CSV | ❌ | ✅ CSV |
| 广告 | 有 | 有（Pro付费去广告） | 无 | 有 | 无 |
| 价格 | 免费+广告 | 免费+内购 | 免费 | 免费+内购 | 免费 |
| 视觉风格 | 传统商务 | 简洁黑白 | 清新简约 | 卡通可爱 | 春日绿色自然 |
| 30日留存 | ~12% | ~18% | ~20% | ~25% | 目标 ≥ 40% |

#### 2.2.2 竞品的三个关键缺陷

1. **功能溢出**：随手记功能多但入口深，用户需要 8 步才能记一笔
2. **统计无洞察**：大多数工具只展示图表，不告诉用户「这意味着什么」
3. **缺乏情感设计**：冰冷的数字让人焦虑，记账变成负担而非习惯

#### 2.2.3 我们的差异化策略

| 维度 | 策略 |
|------|------|
| 体验 | 5 秒记账 + 首页零操作获取全貌 |
| 情感 | 春日绿色主题，传递生机与舒适感，消除记账的焦虑感 |
| 洞察 | 不只展示数字，告诉用户「较上月少花了 12%」这类信息 |
| 信任 | 离线优先 + 数据导出，用户完全掌控 |

### 5.2 功能详细设计

#### F1. 快速记账（P0）

**用户故事**

> 作为小明，我希望在买完午餐后能快速记录这笔消费，不需要思考、不需要填备注、不要超过 5 秒。

**验收标准**

| 编号 | 条件 | 预期结果 |
|------|------|---------|
| AC-01 | 用户点击首页「+」按钮 | 弹出记账面板，数字键盘自动唤起，焦点在金额输入框 |
| AC-02 | 用户输入金额后点击分类 | 分类选中态高亮，显示勾选标记 |
| AC-03 | 用户未选择分类直接保存 | 使用上次选择的分类（首次默认「其他」） |
| AC-04 | 用户点击保存 | 面板收起，首页数据实时刷新，toast 提示「已记录」 |
| AC-05 | 金额为空时点击保存 | 按钮置灰不可点击，金额输入框抖动提示 |
| AC-06 | 金额超过 ¥99,999,999.99 | 拒绝输入，toast 提示「金额超出范围」 |
| AC-07 | 用户选择未来日期 | 未来日期不可选（日历中置灰） |
| AC-08 | 用户不选日期直接保存 | 默认使用当天日期 |
| AC-09 | 用户输入备注后保存 | 备注正常保存，列表中展示 |
| AC-10 | 从点击「+」到保存成功 | 总耗时 < 5 秒（不含用户犹豫时间） |

**状态处理**

| 状态 | 处理方式 |
|------|---------|
| 加载中 | 不适用（本地操作，瞬时响应） |
| 保存成功 | 面板关闭，首页数据实时刷新，绿色 toast「✓ 已记录」 |
| 保存失败 | 红色 toast「记录失败，请重试」，面板保持打开不丢失输入内容 |
| 金额为空 | 保存按钮置灰（opacity: 0.4），不可点击 |
| 金额超限 | 输入框下方红色文字提示「单笔金额不能超过 ¥99,999,999.99」 |
| 网络断开 | 不影响（纯本地存储），正常保存 |

**边界情况**

| 场景 | 处理规则 |
|------|---------|
| 金额输入「0.00」 | 视为无效金额，保存按钮置灰 |
| 金额输入「.5」 | 自动补全为「0.5」 |
| 金额输入「000123」 | 自动去前置零，保留「123」 |

## 6. 交互设计规范

### 6.1 设计原则

| 原则 | 说明 | 实例 |
|------|------|------|
| 即时反馈 | 每次操作 200ms 内给予反馈 | 保存按钮点击后立即有触觉反馈 + toast |
| 容错设计 | 关键操作可撤销 | 删除记录支持 3 秒内撤回 |
| 渐进呈现 | 先展示核心信息，详情感兴趣再展开 | 首页显示 Top3 分类，点击展开全部 |

## 8. 数据模型

### 8.1 实体关系图

```
┌──────────────┐       ┌──────────────┐
│   Category   │       │   Expense    │
├──────────────┤       ├──────────────┤
│ id (PK)      │←──────│ category_id  │
│ name         │  1:N  │ amount       │
│ icon         │       │ note         │
│ color        │       │ date         │
│ is_preset    │       │ created_at   │
│ sort_order   │       │ updated_at   │
│ created_at   │       │ deleted_at   │  ← 软删除
└──────────────┘       └──────────────┘
                               │
                       ┌───────┴───────┐
                       │    Budget     │
                       ├───────────────┤
                       │ id (PK)       │
                       │ month         │  ← '2026-04'
                       │ category_id   │  ← nullable（null = 总预算）
                       │ amount        │
                       │ created_at    │
                       │ updated_at    │
                       └───────────────┘
```

### 8.2 表结构详细设计

#### Category 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| name | TEXT | NOT NULL, UNIQUE | 分类名称 |
| icon | TEXT | NOT NULL | Emoji 图标 |
| color | TEXT | NOT NULL | 色值，如 '#7EC850' |
| is_preset | INTEGER | NOT NULL, DEFAULT 0 | 是否预设分类（0=自定义, 1=预设） |
| sort_order | INTEGER | NOT NULL, DEFAULT 0 | 排序权重，越小越靠前 |
| created_at | TEXT | NOT NULL | 创建时间 ISO8601 |
| deleted_at | TEXT | DEFAULT NULL | 软删除标记 |

#### Expense 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| category_id | INTEGER | FK → Category.id, NOT NULL | 所属分类 |
| amount | REAL | NOT NULL, > 0 | 金额（正数） |
| note | TEXT | DEFAULT '' | 备注 |
| date | TEXT | NOT NULL | 消费日期 'YYYY-MM-DD' |
| created_at | TEXT | NOT NULL | 记录创建时间 ISO8601 |
| updated_at | TEXT | NOT NULL | 最后更新时间 ISO8601 |
| deleted_at | TEXT | DEFAULT NULL | 软删除标记 |

**索引：**
- `idx_expense_date` ON `date`（按日期查询是最高频操作）
- `idx_expense_category` ON `category_id`（按分类筛选）
- `idx_expense_date_category` ON `(date, category_id)`（复合查询）

#### Budget 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| month | TEXT | NOT NULL | 预算月份 'YYYY-MM' |
| category_id | INTEGER | FK → Category.id, NULLABLE | NULL = 月度总预算 |
| amount | REAL | NOT NULL, > 0 | 预算金额 |
| created_at | TEXT | NOT NULL | 创建时间 |
| updated_at | TEXT | NOT NULL | 更新时间 |

**唯一约束：** `UNIQUE(month, category_id)` — 同一月份同一分类只能有一条预算

### 8.3 数据流

```
用户操作
  │
  ├── 记一笔 ──→ Expense 表 INSERT
  │               │
  │               ├──→ 首页总额实时重算（SUM WHERE month=当前月）
  │               ├──→ 分类占比实时重算（GROUP BY category_id）
  │               ├──→ 预算进度实时重算（SUM vs Budget.amount）
  │               └──→ 图表数据实时更新
  │
  ├── 删记录 ──→ Expense 表 SOFT DELETE（deleted_at = now）
  │               └──→ 同上，所有统计数据重算
  │
  ├── 改记录 ──→ Expense 表 UPDATE
  │               └──→ 同上，所有统计数据重算
  │
  ├── 新增分类 → Category 表 INSERT
  │
  └── 设预算  ──→ Budget 表 UPSERT
```

### 8.4 统计计算规则

| 统计项 | 计算逻辑 |
|--------|---------|
| 本月支出 | `SUM(amount) FROM Expense WHERE strftime('%Y-%m', date) = '当前月' AND deleted_at IS NULL` |
| 分类占比 | `SUM(amount) GROUP BY category_id` / 本月总支出 × 100% |
| 环比变化 | `(本月支出 - 上月支出) / 上月支出 × 100%` |
| 预算进度 | `本月支出 / 预算金额 × 100%` |
| 7天趋势 | 最近 7 天（含今日）每天 `SUM(amount) GROUP BY date` |
| 日均支出 | `本月支出 / 本月已过天数` |

## 9. 技术架构

### 9.1 架构概览

```
┌─────────────────────────────────────────┐
│              前端（Frontend）             │
│  React + TypeScript + Vite              │
│  ├── UI 组件层（Components）             │
│  ├── 状态管理（Zustand）                 │
│  ├── 路由管理（React Router）            │
│  └── 图表渲染（ECharts / Recharts）      │
├─────────────────────────────────────────┤
│           数据访问层（Data Layer）         │
│  ├── SQLite 封装（sql.js / better-sqlite3）│
│  ├── 数据迁移（Migration）               │
│  └── 缓存策略（内存缓存 + 预计算）        │
├─────────────────────────────────────────┤
│           存储层（Storage）               │
│  ├── 本地 SQLite 数据库                  │
│  └── 导出文件（CSV）                     │
└─────────────────────────────────────────┘
```

### 9.2 技术选型

| 层级 | 方案 | 选型理由 |
|------|------|---------|
| 开发框架 | React 18 + TypeScript | 生态成熟、类型安全 |
| 构建工具 | Vite 5 | 开发体验快，HMR 即时响应 |
| UI 组件库 | 自研 + Radix UI | 轻量，可定制，符合春日设计风格 |
| 状态管理 | Zustand | 比 Redux 轻，API 直观 |
| 路由 | React Router v6 | 标准方案 |
| 图表 | Recharts | 声明式、React 原生、轻量 |
| 本地数据库 | sql.js（Web）/ 原生 SQLite（Mobile） | 无需后端，纯客户端 |
| 样式方案 | Tailwind CSS + CSS Variables | 原子化 + 主题变量灵活切换 |
| 测试 | Vitest + Playwright | 单元测试 + E2E |
| 代码规范 | ESLint + Prettier + Husky | 提交前自动检查 |

### 9.3 性能策略

| 策略 | 说明 |
|------|------|
| 预计算 | 首页统计数值在每次数据变更时重新计算并缓存，打开首页即读缓存 |
| 虚拟列表 | 当记录超过 200 条时启用虚拟滚动，仅渲染可视区域 |
| 图表降级 | 数据量超过 5000 条时，图表自动切换为按月聚合（而非按天） |
| 懒加载 | 统计页图表在 Tab 切换到时才首次渲染 |
| 本地数据库 | SQLite 查询 < 50ms（10000 条内），无需网络延迟 |
